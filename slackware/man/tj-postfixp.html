1. Overview

Postfix é un MTA suddiviso in diversi programmi che svolgono una azione specifica ed un programma principale che si occupa di chiamare il programma giusto quando necessario e di gestire le varie azioni. Solo il programma master necessita di privilegi elevati e va lanciato da root, gli altri invece possono girare con privilegi bassi usando un utente creato appositamente.
Ogni programma che compone questo complesso sistema agisce in modo autonomo senza avere processi figli, in questo modo si cerca di evitare relazioni tra i processi che potrebbero condizionare il comportamento dei child in caso il processo genitore sia compromesso.
Mentre i protocolli per il mail-retriving (POP3/IMAP) obbligatoriamente autenticano l'utente, l'SMTP non prevede di default procedure di autenticazione, permettendo l'invio di email ad utenti sconosciuti, inoltre per default le transazioni fra MUA-MTA sono in chiaro. Utilizzando un modulo delle librerie di autenticazione SASL é possibile realizzare la procedura di autenticazione, mediante la compilazione di TLS in Postfix viene cifrata la connessione proteggendo password e comprovando la propria identità con certificati. 
- Il MUA chiede relay per l'invio a MTA

- TLS risponde con certificato Server, Authority e richiesta password

- MUA passa utente+password a SASL,framework autenticazione verifica

- password e mail rimangono cifrate unicamente sul percorso MUA-MTA
http://www.postfix.org/docs.html


2. Tecnologie utilizzate

SASL
SASL significa 'Simple Authentication and Security Layer'. È standardizzato da IETF (Internet Engineering Taskforce). SASL è utililzzato da server di rete per gestire le richieste di autenticazione dei clients. Dispone di 3 eseguibili: 2 per gestire l'autenticazione del database utenti tramite librerie di Berkeley, ed un deamon 'saslauthd' che esegue le richieste. È possibile utilizzare diversi framework (backend) di autenticazione, decisi in fase di compilazione: base dati proprietaria (Berkeley DB) oppure tramite PAM a Mysql, Kerberos ecc., oppure la base dati di sistema /etc/shadow o perfino /etc/passwd mediante saslauthd.
http://asg.web.cmu.edu/sasl

TLS
Il protocollo SSL (Secure Socket Layers) fu introdotto da Netscape poi evoluto in TLS (Transportation Layer Security) standardizzato dalla RFC2246. Consente la cifratura e l'autenticazione forte, intesa come 'sicurezza che le due parti comunicanti siano correttamente identificate e che la comunicazione non possa essere alterata'.
Postfix/TLS non realizza in stand-alone il protocollo TLS, ma utilizza il package OpenSSL per adempiere a questo compito. L'integrazione del protocollo TLS al protocollo SMTP (Simple Mail Transport Protocol) é descritto dalla RFC2487. A differenza della prima concezione di SSL come contenitore di una comunicazione normale (STUNNEL) il protocollo TLS viene integrato completamente dentro SMTP: all'inizio della negoziazione con EHLO, il server fornisce il supporto TLS mediante il comando STARTTLS, a sua volta il client può rispondere ed avviare la sessione cifrata.
Il modulo TLS viene fornito come patch per il sorgente di Postfix, é importante ottenere la patch in versione coerente a quella di OpenSSL installata.
http://www.aet.tu-cottbus.de/personen/jaenicke/postfix_tls

OpenSSL
OpenSSL è una realizzazione in forma di software libero dei protocolli SSL/TLS (Secure socket layer e Transport layer security) per la certificazione e la comunicazione cifrata. Inizialmente, il progetto si chiamava SSLeay, ma da quando l'autore originale lo ha dovuto interrompere, questo è stato ripreso da un gruppo indipendente che lo ha ribattezzato in OpenSSL. Si compone di alcune librerie che permettono di incorporare le funzionalità dei protocolli SSL/TLS all'interno di programmi di comunicazione, oltre a una serie di programmi di servizio per la gestione delle chiavi e dei certificati, arrivando eventualmente anche alla gestione di un'autorità di certificazione[1]
http://www.openssl.org


3. SASL - compilazione

Il presente 'configure' e seguenti prevede supporto solo a saslauthd e relativa lettura degli account su /etc/shadow. Ragioni di compatibilità client chiedono supporto alle features [ login - plain - digest ] 
default prefix: /usr/local, presentato quello per package build
Slackware flavour.

./configure \
--prefix=/usr/local/src/cyrus-sasl-2.1.19/PACK/usr/local \ 
--disable-anon \
--disable-krb4 \
--disable-otp \
--disable-cram \
--disable-gssapi \
--disable-java \
--enable-static \
--enable-login \
--enable-plain \
--enable-digest \
--enable-sample \
--with-saslauthd=/var/run/saslauthd \
--with-dblib=none \
--without-pam \
--with-openssl=/usr/include/openssl \
--with-plugindir=/usr/local/lib/sasl2

make
make install
Operazioni di compilazione finali: 
cd saslauthd, make testsaslauthd ,cp testsaslauthd /usr/local/bin

Posizione delle librerie sovente richiesta: 
ln -s /usr/local/lib/sasl2/ /usr/lib/sasl2

Configurazione framework utilizzata da postfix 
echo 'pwcheck_method: saslauthd' > /usr/lib/sasl2/smtpd.conf
chmod 640 /usr/lib/sasl2/smtpd.conf
chown root.postfix /usr/lib/sasl2/smtpd.conf
CAVE: se manca o errato non funziona l'autenticazione, senza però msg d'errore relativi significativi.
Il deamon verrà attivato dal comando seguente 
saslauthd -a shadow

Script di startup /etc/rc.d/rc.saslauthd Slackware flavour. 
#!/bin/bash
#
# Cyrus startup script
case "$1" in
start)
# Starting SASL saslauthdaemon 
/usr/local/sbin/saslauthd -a shadow
;;

stop)
# Stopping SASL saslauthdaemon
killall saslauthd
;;
*)
echo "Usage: $0 {start|stop}"
exit 1
;;
esac

Da aggiungere ad /etc/rc.d/rc.inetd.2 
# Start SASL (the saslauthd daemon ):
if [ -x /etc/rc.d/rc.saslauthd ]; then
  . /etc/rc.d/rc.saslauthd start
fi


4. TLS - integrazione

Ottenibile negli addon di postfix.org, é necessario installare la patch TLS prima della creazione del makefile. Verificare la versione corrispondente di OpenSSL
Decomprimere la patch 'pfixtls.diff' nella src di postfix 
 patch -p1 < pfixtls.dif
L'opzione di default prevede 'patch -p0', ma possibili problemi possono nascere da una numerazione di versione superiore del sorgente Postfix.


5. Compilazione Postfix

Compilazione Posfix con abilitazione TLS e SASL su Slack 10, gcc 2.3.4, causa linker, le opzioni -R -rpath, previste di default, non vengono riconosciute, workaround: '-Wl,-rpath,' CAVE: ai blank non richiesti, generano fastidiosi errori 
make makefiles \
CCARGS=" -DUSE_SASL_AUTH -DUSE_SSL \
-I/usr/local/include/sasl -I/usr/include/openssl" \
AUXLIBS="-L/usr/lib -Wl,-rpath,/usr/lib -lssl -lcrypto \
 -L/usr/local/lib -lsasl2 -Wl,-rpath,/usr/local/lib"
 
 make

 make install 
 chiederà path di installazione

Creazione dei diritti di accesso Postix: l'installer di Slackware esegue per voi l'operazione

6. Certificati

generazione semplice, solo cifratura, fake-identity con l'utility OpenSSL: CA.pl 
CA.pl -newca		#Crea l'authorithy e la dir demoCA

openssl req -new -nodes -keyout mailkey.pem \
-out mailreq.pem -days 365

openssl ca -out mail_signed_cert.pem -infiles mailreq.pem
in /etc/postfix sono necessari 3 certificati in formato .pem: 
- cacert.pem            # Certification Authority del server
- mailkey.pem           # Chiave privata (chown root e chmod 400)
- mail_signed_cert.pem  # Chiave firmata, pubblica rilasciata ai client


7. Configurazioni

Minima configurazione Postfix /etc/postfix/main.cf necessaria al sistema SASL/TLS
Postfix generico, senza relay, rete privata 
# layout SLACKWARE (da adattare)
queue_directory = /var/spool/postfix
daemon_directory = /usr/libexec/postfix
mail_owner = postfix
myorigin = $myhostname
inet_interfaces = all
mydestination = $myhostname, localhost.$mydomain, localhost
mynetworks = 127.0.0.0/8, 192.168.1.0/24
alias_database = hash:/etc/aliases
sender_canonical_maps = hash:$ /etc/sender_canonical
smtpd_banner = $myhostname ESMTP $mail_name (Slackware/GNU)

Gestione Aliases

Il file /etc/postfix/aliases, Contiene la mappatura di un un utente di posta elettronica con un utente di sistema. Questo importante per 2 motivi: è fondamentale che esistano gli alias per l'utente postfix,per il corretto funzionamento, e per root, sarebbe preferibile non ricevesse mail direttamente). Il secondo motivo é dato dall'esistenza di alcuni account di ruolo che vanno creati nel rispetto della netiquette, ad esempio postmaster, info, abuse, ecc; dal momento che, nella pratica, spesso è un solo utente a svolgere le funzioni, è possibile fare in modo che le relative e-mail vengano automaticamente girate nella mailbox dell'utente.
etc/postfix/aliases conterrà quindi le voci: 
  postmaster:   tom
  abuse:        tom
  info:	        tom
  postfix:      root
  root:	        tom
È possibile utilizzare anche alias ricorsivi (Postfix->root->tom).Bisognerà utilizzare il comando /usr/bin/newaliases per rendere effettive le modifiche con conseguente creazione del file aliases.db.
Per far corrispondere gli indirizzi email agli utenti locali postfix utilizza il file sender_canonical per riscrivere gli indirizzi in uscita, che contiene gli indirizzi che verranno inseriti nel campo from per ogni utente al posto di utente@vostra.macchina 
# Synopsis sender_canonical

root    tom@domain.top
mark    tom@domain.top
tom     tom@domain.top
Utilizzando il comando postmap /etc/postfix/sender_canonical verranno rese effettive le modifiche con conseguente creazione del file sender_canonical.db.

Aggiunta SASL 
# SASL
smtpd_sasl_auth_enable = yes
broken_sasl_auth_clients = yes
smtpd_sasl_security_options = noanonymous
smtpd_recipient_restrictions = permit_mynetworks, permit_sasl_authenticated,reject_unauth_destination

Aggiunta TLS con posizione certificati 
# TLS
smtpd_use_tls = yes
smtpd_tls_key_file = /etc/postfix/mailkey.pem
smtpd_tls_cert_file = /etc/postfix/mail_signed_cert.pem
smtpd_tls_CAfile = /etc/postfix/cacert.pem
Dopo ogni modifica dovete riavviare postfix con postfix reload.
